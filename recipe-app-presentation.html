1. アプリ概要 — 飲食業務をひとつにまとめる

本アプリは「レシピ」「原価」「在庫」「仕込み」「発注」を統合管理する Web アプリケーションです。PC・タブレット・スマホ対応。

日本語 / 英語 / フランス語 / イタリア語の AI 自動翻訳。

3. レシピ管理機能

入力モード

  • 通常モード: 一般料理用。材料名 / 分量 / 単位 / 単価を入力し、仕入れマスタから原価が自動入力。
  • パン Baker's % モード: 粉を 100% としてその他の材料を比率で管理。狙いの総重量 (g) を入れると全材料が再計算される。

整理と共有

  • ドラッグ&ドロップ並び替え（PC: マウス、スマホ: 1秒長押し）
  • タグ・カテゴリ・店舗・コースで多軸フィルタリング
  • 共有レシピの安全設計: 他人のレシピを編集して保存すると、元を壊さないよう 自動的にコピー が作成される
  • 削除はゴミ箱経由の2段階（誤操作対策）

4. パン用 Baker's % モード

粉重量を基準（100%）として、他の材料を粉に対する百分率で記録するパン業界標準の方式。

各材料量 (g) = 粉総量 (g) × その材料の Baker's %
総生地量 (g) = 粉 × (1 + Σ 各材料の比率)

具体例：バゲット生地

材料Baker's %粉1000gのとき粉1500gのとき
強力粉100%1000 g1500 g
水68%680 g1020 g
塩2%20 g30 g
イースト0.4%4 g6 g

アプリでは「狙いの総重量 (g)」を入れると逆算で粉量・全材料を自動計算。日々レシピをスケールさせるパン職人にとって不可欠な機能。

5. 原価計算（自動化）

仕入れマスタ（CSV）と材料マスター（unit_conversions）が単価と内容量を保持。レシピ入力時に名称マッチで自動取得。

kg/L単価への正規化（桁違い防止の核心）

単位あたり単価 = lastPrice ÷ packetSize
例: 25,000 g / 袋, 10,900 円 / 袋 → 0.436 円 / g

税率の自動判定

  • 区分が alcohol（アルコール）/ supplies（備品）→ 10%
  • それ以外（食材）→ 8%

歩留まり（可食率）

魚・肉などの 100% 未満材料は cost × 1/yield で実質原価を上げて計算。原価欄下に「歩留まり: 50%」のように表示。

6. カテゴリ別原価

レシピ内の材料は グループ（ソース、生地、仕上げ等） に分けられ、カテゴリ単位で原価を確認・上書きできます。

ロジック（categoryCostOverrideService.js）

const buildGroupEntry = (categoryKey, categoryName, ingredients, multiplier) => {
  const costTaxIncluded = ingredients.reduce(
    (sum, item) => sum + getScaledCostTaxIncluded(item, multiplier),
    0
  );
  return { categoryKey, categoryName, ingredients, costTaxIncluded };
};

// 上書き（override）があればそちらを優先
const total = categories.reduce((sum, c) => {
  const overridden = overrideMap.get(c.categoryKey);
  return sum + (Number.isFinite(overridden) ? overridden : c.costTaxIncluded);
}, 0);

「ムースを 20g 使ったときの原価」のようなカテゴリ使用量シミュレーションがここから派生します。後述する合成レシピの計算とも連携。

8. 仕込みカレンダー & 発注リスト

仕込みカレンダー（Planner）

  • 日付にレシピをドラッグ&ドロップで予定登録
  • 予定の日付移動・削除、複数レシピの同日管理、オーバーワーク警告

発注リスト（OrderList）— 2割ルール

必要量 = Σ（仕込み予定のレシピごとに必要な材料量）
残在庫 = max(0, 在庫 − 必要量)
発注対象 ⇔ 残在庫 < 1発注単位 × 0.2（2割）
発注量 = ⌈不足量 ÷ packetSize⌉ （袋・本単位で切り上げ）

例: ビネガー 1本=750ml、2割=150ml を切ったら発注対象。発注量は ml ではなく 1本として表示。

9. 単位の優先順位（運用の核心）

原価・在庫・発注の桁違いはほぼ「単価の意味」と「単位の優先順位」の崩れで起きる。本アプリは次の階層で正規化します。

  1. 材料マスター（unit_conversions） — packetSize, packetUnit, lastPrice。最優先。
  2. 元の単位（CSV）上書き（csv_unit_overrides） — 「袋・本・箱」など発注単位の補正。
  3. 仕入れCSV（仕入れマスタ） — 単価と業者情報。
  4. 手入力 — レシピ・在庫の最終フォールバック。
在庫画面で材料マスターがある場合、単価は lastPrice / packetSize に正規化、本/袋を g/ml に換算。これにより「在庫金額が壊れない」設計。

RLS（Row-Level Security）

CREATE POLICY recipes_update_own_or_admin
  ON recipes FOR UPDATE TO authenticated
  USING (
    EXISTS (
      SELECT 1 FROM profiles
      WHERE id = auth.uid() AND role = 'admin'
    )
    OR tags @> ARRAY['owner:' || auth.uid()::text]
    OR EXISTS (
      SELECT 1 FROM profiles
      WHERE id = auth.uid()
        AND display_id IS NOT NULL
        AND tags @> ARRAY['owner:' || display_id]
    )

  WITH CHECK (

    AND jsonb_array_length(coalesce(ingredients, '[]'::jsonb)) <= 200
  );

13. 合成レシピとは何か

複数のレシピを組み合わせて作る商品（惣菜パン、プレート料理、デザート盛り合わせ等）について、「どのレシピを何 g 使うか」を入力すると原価・原価率・粗利が一発で出る機能。

解決する課題

  • ベーカリーで「バゲット + ポテサラ + バター」のような複合商品の原価がどんぶり勘定になる
  • 1つのソース仕込みから一部だけ使う場合の原価計算が手計算で煩雑
  • レシピごとに「総出来上がり量」と「使用量」が違うため単位原価への分解が必要

提供する価値

  • 商品設計時に 原価率を見ながら販売価格を決定できる
  • 組み合わせを保存→再編集→再保存できるので、季節メニューの試作管理に最適

15-① 計算ロジック: 出来上がり量の集計

レシピは材料の単位がバラバラ（g, kg, ml, cl, l, 個, 本…）。これを「総出来上がり量」に統一する関数が computeBatchStats()。

const computeBatchStats = (ingredients) => {
  let totalWeightGrams = 0;
  let totalVolumeMl   = 0;
  for (const item of ingredients) {
    const qty  = parseFloat(item.quantity);
    const unit = normalizeUnit(item.unit || '');
    if (unit === 'g')             totalWeightGrams += qty;
    if (unit === 'kg')            totalWeightGrams += qty * 1000;
    if (unit === 'ml' || unit === 'cc') totalVolumeMl += qty;
    if (unit === 'cl')            totalVolumeMl    += qty * 10;
    if (unit === 'l')             totalVolumeMl    += qty * 1000;
  }
  const defaultBatchAmount = totalWeightGrams + totalVolumeMl; // 1ml ≈ 1g 概算
  const defaultUnit        = totalWeightGrams > 0 ? 'g' : 'ml';
  return { totalWeightGrams, totalVolumeMl, defaultBatchAmount, defaultUnit };
};

★ 個 / 本などの個数単位は意図的に除外。重量と液量が混在する場合は 1 ml ≈ 1 g の概算で合算するのが現場的な妥協点。手動調整も可能。

15-② 計算ロジック: 単位原価と使用原価

各レシピを「1g（または1ml）あたりいくら」に分解し、使用量を掛けて使用原価を出します。

単位原価 = レシピの税込総原価 ÷ 総出来上がり量
使用原価 = 単位原価 × 使用量
合成原価合計 = Σ 各レシピの使用原価

該当コード（RecipeCompositeCostCalculator.jsx）

const unitCost = batch > 0
  ? (totalCostTaxIncluded / batch)
  : NaN;
const lineCost = (Number.isFinite(unitCost) && use >= 0)
  ? (unitCost * use)
  : NaN;

// 合計
const totalCompositeCost = currentLine.lineCost
  + sum(otherLines.map(l => l.lineCost));

税込原価は computeRecipeTotalCostTaxIncluded()（categoryCostOverrideService）から取得。カテゴリ別の原価上書きがあれば自動的に反映。

15-③ 計算ロジック: 販売価格・粗利・原価率

1個あたり原価 (税込) = 合成原価合計
予想売上 = 販売価格 × 販売数
粗利益 = 予想売上 − (1個あたり原価 × 販売数)
合成原価率 = 1個あたり原価 ÷ 販売価格

2026/05 修正のポイント

旧版では原価率に販売数が混入し「販売数を変えると原価率が動く」という不合理が発生。新版は「1個あたりの原価率は販売数に依存しない」という当たり前の数学に修正。 
// totalCompositeCost は「1個あたりの原価」
// したがって costRate は count に依存しない
const costRate = hasPrice && Number.isFinite(unitCost)
  ? (unitCost / price) * 100
  : NaN;

17. 保存・編集フロー

保存（compositeRecipeService.createSet）

  1. recipe_composite_sets にヘッダー INSERT → 採番された id を取得
  2. 計算機の rows を recipe_composite_set_items へ一括 INSERT（composite_set_id に親IDを差し込む）
  3. 失敗時は例外を Toast で通知。RLS違反は 401/403 として現れる

更新（updateSet）

  1. ヘッダーを UPDATE（updated_at = now()）
  2. 明細を 一旦全削除してから再 INSERT（差分更新ではなく置き換え）
  3. 明細の並び順は sort_order で保持

編集ページ（RecipeCompositeCostEditPage）

保存済み合成レシピを開くと、getSetDetail でヘッダー＋明細を取得し、recipeService.getRecipe でベースレシピも引き直す。これにより「— 表示」で原価が出ない問題を解消。

18. カテゴリ原価との連携

合成原価の計算で参照する「各レシピの税込総原価」は、単純な合算ではなく カテゴリ別原価の上書きを尊重します。

処理フロー

  1. 合成計算機が各レシピをロードする際、categoryCostOverrideService.fetchByRecipeId で recipe_category_cost_overrides から上書きMapを取得
  2. computeRecipeTotalCostTaxIncluded(recipe, overrideMap) で総原価を再計算
  3. カテゴリごとに override があればそれを採用、無ければ元材料から再計算
例: ムースの「ベース」カテゴリだけ業者変更で原価が下がった場合、ベースのみ上書きを設定すれば、合成レシピの計算にもその下げた原価が自動反映。

仕入れ価格の振れに対して柔軟に追従できる、運用上きわめて重要な設計。

  • 編集ページで — が出る問題を解消（保存後のレシピ詳細を再取得）

UX改善

  • 固定ベース廃止: 検索結果から連続して組み合わせ追加可能、先頭レシピも入れ替え・削除可
  • 「ベースレシピ」プルダウンと「+ 組み合わせを追加」ボタンを廃止し、検索カード中心の操作に統一
  • 保存済み一覧のカードデザインと視認性向上、トップページへの導線追加

20. シェフ・パン職人視点での価値

1. レシピを科学的に分解できる

「ソース 1500 g 仕込み中 40 g 使用」のような状況を、感覚ではなく 1 g 単位で原価化。フランス料理のコース構成、惣菜パンの構成それぞれで活躍。

2. メニュー設計時の意思決定が速い

販売価格を変えると原価率がリアルタイムに動く。FoodCost 30% を切る販売価格を逆算しながらメニューを組める。

3. 季節商品の試作管理

春の限定パン、秋のテリーヌ、クリスマスのデザートなど 同じ「合成レシピ」をシーズンごとに保存しておけば、翌年の同時期にすぐ呼び出せる。

4. パン職人の「組み合わせ商品」管理に最適

バゲット生地 + クリーム + フルーツ といった構成を、使用量別の単位原価で正確に把握。仕込みロス削減にも直結。

23. 課題と今後の展望

現状の制約

  • 個・本などの個数単位を含むレシピは「総出来上がり量」に含まれず、合成計算が困難
  • 合成レシピを「他者と共有」する仕組みは未実装（個人ライブラリのみ）
  • 合成レシピを 1つの新レシピとして登録する自動化はない
  • 履歴・バージョン管理が無く、過去の構成変更を遡れない

提案する改善案

  1. 個数単位の1個あたり重量をレシピに保持し、自動換算可能にする
  2. 合成レシピ → 通常レシピへの「展開保存」機能（材料を全部足し合わせて1レシピ化）
  3. カテゴリ原価 override の有効期限を設定（仕入れ価格が変わったら自動失効）
  4. 合成レシピの店舗共有（is_shared フラグ追加）
  5. 変更履歴テーブルで「2026年4月の構成」「5月の構成」を比較可能にする

24. まとめ

アプリ全体

レシピ・原価・在庫・仕込み・発注を統合管理する飲食業向け Web アプリ。React + Supabase + RLS で安全とパフォーマンスを両立。

合成レシピの本質

各レシピを「1 g（1 ml）あたりいくら」に分解し、使用量を掛け合わせて合計するという、シンプルだが現場で価値の高い計算機。

計算式の核

単位原価 = 税込原価 ÷ 出来上がり量／使用原価 = 単位原価 × 使用量／原価率 = 1個原価 ÷ 販売価格（販売数に依存しない）

データ設計

ヘッダー＋明細の2テーブル＋RLS。recipe_composite_sets / recipe_composite_set_items でユーザーごとに完全分離。

2026/05の進化

保存・編集・一覧化＋計算式の修正＋検索カード中心UIへの再設計。

シェフへの提供価値

感覚ではなく数式で「使った分だけの原価」を即時把握。メニュー設計と季節商品試作管理に直結。

フランス料理 / ベーカリーの現場では「仕込みは多めに、使うのは一部」という構造が日常。本機能はその構造を科学的に数値化するためのツールです。
